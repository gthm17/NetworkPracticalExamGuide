TCP — Transmission Control Protocol Connection-Based

TCP is like a phone call — you establish a connection first, then talk, then hang up. Reliable, ordered delivery guaranteed.

How TCP Works — The Flow
1socket() — Both server & client create a socket (endpoint)
↓ Server only
2bind() — Server attaches socket to a port (3003)
3listen() — Server starts waiting for connections
4accept() — Server accepts incoming client
↓ Client only
5connect() — Client connects to server IP:port
↓ Both
6send() / recv() — Exchange data
7close() — Terminate connection
SERVER
socket() → bind() → listen() → accept()
↕3-Way Handshake (SYN, SYN-ACK, ACK)
CLIENT
socket() → connect()
↕send() / recv() both directions
Key Socket Args
AF_INET
IPv4 address family
SOCK_STREAM
TCP — stream based, reliable
INADDR_ANY
Accept connections on any interface
TCP Server SERVER

Server binds to port, listens, accepts client, then does recv → send.

// TCP SERVER — Networks Lab
#include <sys/socket.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <netinet/in.h>

int main() {
    int sock_desc, temp_sock_desc, k, len;
    char buf[100];
    struct sockaddr_in server_addr, client;

    // Step 1: Create socket
    sock_desc = socket(AF_INET, SOCK_STREAM, 0);
    if (sock_desc == -1)
        printf("Error in socket creation\n");

    // Step 2: Set server address
    server_addr.sin_family = AF_INET;
    server_addr.sin_addr.s_addr = INADDR_ANY;
    server_addr.sin_port = htons(3003);

    // Step 3: Bind socket to address
    k = bind(sock_desc, (struct sockaddr*)&server_addr, sizeof(server_addr));
    if (k == -1)
        printf("Error in binding\n");

    // Step 4: Listen for connections
    k = listen(sock_desc, 5);
    if (k == -1)
        printf("Error in listening\n");

    // Step 5: Accept a client
    len = sizeof(client);
    temp_sock_desc = accept(sock_desc, (struct sockaddr*)&client, &len);
    if (temp_sock_desc == -1)
        printf("Error in accept\n");

    // Step 6: Receive message from client
    k = recv(temp_sock_desc, buf, 100, 0);
    printf("Message from client: %s\n", buf);

    // Step 7: Send reply
    printf("Enter data to send: ");
    fgets(buf, 100, stdin);
    send(temp_sock_desc, buf, 100, 0);

    close(temp_sock_desc);
    close(sock_desc);
    return 0;
}
🔑 REMEMBER
Server uses TWO sockets: sock_desc (listening) and temp_sock_desc (actual communication with client). accept() returns the second one.
TCP Client CLIENT

Client just creates socket, connects, then does send → recv. No bind/listen/accept needed!

// TCP CLIENT — Networks Lab
#include <sys/socket.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <netinet/in.h>

int main() {
    int sock_desc, k;
    char buf[100];
    struct sockaddr_in client;

    // Step 1: Create socket (SAME as server)
    sock_desc = socket(AF_INET, SOCK_STREAM, 0);
    if (sock_desc == -1)
        printf("Error in socket creation\n");

    // Step 2: Set server address to connect TO
    client.sin_family = AF_INET;
    client.sin_addr.s_addr = INADDR_ANY;
    client.sin_port = htons(3003); // SAME port as server

    // Step 3: Connect to server (replaces bind+listen+accept)
    k = connect(sock_desc, (struct sockaddr*)&client, sizeof(client));
    if (k == -1)
        printf("Error in connecting\n");

    // Step 4: Send message first (client initiates)
    printf("Enter data to send: ");
    fgets(buf, 100, stdin);
    send(sock_desc, buf, 100, 0);

    // Step 5: Receive reply from server
    k = recv(sock_desc, buf, 100, 0);
    printf("Message from server: %s\n", buf);

    close(sock_desc);
    return 0;
}

UDP — User Datagram Protocol Connectionless

UDP is like sending a letter — no handshake, no guarantee. Faster but unreliable. This version uses command-line args for IP/port and is one-way: client sends, server only receives. Key headers: <arpa/inet.h> (client) and <unistd.h> (server).

TCP vs UDP Quick Diff
FeatureTCPUDP
Socket typeSOCK_STREAMSOCK_DGRAM
Connectionconnect()No connect
Sendsend()sendto()
Receiverecv()recvfrom()
Reliable✅ Yes❌ No
Server acceptaccept()No accept
UDP Flow (New Version)
1socket(AF_INET, SOCK_DGRAM, 0) — both sides
↓ Server only
2bind() — server binds to argv[1] port
3recvfrom() — server waits & receives
↓ Client only
4sendto() — client sends to argv[1] IP, argv[2] port
5close() — both sides close socket
UDP Server SERVER

Run as: ./server <Port>. Server binds to the given port, prints Server waiting..., then blocks on recvfrom(). When a datagram arrives it prints it and exits. No reply sent — one-way only.

// UDP SERVER PROGRAM

#include <stdio.h>
#include <string.h>
#include <sys/socket.h>
#include <stdlib.h>
#include <netinet/in.h>
#include <unistd.h>

int main(int argc, char *argv[]) {
    struct sockaddr_in server, client;

    // Check input format
    if (argc != 2) {
        printf("Usage: %s <Port>\n", argv[0]);
        return 0;
    }

    // Create socket
    int sockfd = socket(AF_INET, SOCK_DGRAM, 0);
    if (sockfd == -1) {
        printf("Error in socket()\n");
        return 0;
    }

    // Server details
    server.sin_family = AF_INET;
    server.sin_addr.s_addr = INADDR_ANY;
    server.sin_port = htons(atoi(argv[1]));

    // Bind socket
    if (bind(sockfd,
             (struct sockaddr *)&server,
             sizeof(server)) < 0) {
        printf("Error in bind()!\n");
        return 0;
    }

    char buffer[100];
    socklen_t client_len = sizeof(client);

    printf("Server waiting...\n");

    // Receive message
    if (recvfrom(sockfd,
                 buffer,
                 sizeof(buffer),
                 0,
                 (struct sockaddr *)&client,
                 &client_len) < 0) {
        printf("Error in recvfrom()!\n");
        return 0;
    }

    printf("Got a datagram: %s\n", buffer);

    close(sockfd);
    return 0;
}
🔑 REMEMBER
Server uses argc != 2 — only takes port as arg. atoi(argv[1]) converts the port string to int. socklen_t client_len must be initialised with sizeof(client) before passing its address to recvfrom().
UDP Client CLIENT

Run as: ./client <IP Address> <Port>. Client takes the server IP and port on the command line, uses inet_addr() to convert the IP string, sends one message, then exits. No recvfrom() — one-way only.

// UDP CLIENT PROGRAM

#include <stdio.h>
#include <string.h>
#include <sys/socket.h>
#include <stdlib.h>
#include <netinet/in.h>
#include <arpa/inet.h>

int main(int argc, char *argv[]) {
    struct sockaddr_in server;
    int sockfd;

    // Check input format
    if (argc != 3) {
        printf("Usage: %s <IP Address> <Port>\n", argv[0]);
        return 0;
    }

    // Create socket
    sockfd = socket(AF_INET, SOCK_DGRAM, 0);
    if (sockfd == -1) {
        printf("Error in socket()\n");
        return 0;
    }

    // Server details
    server.sin_family = AF_INET;
    server.sin_port = htons(atoi(argv[2]));
    server.sin_addr.s_addr = inet_addr(argv[1]);

    char buffer[100];

    printf("Enter a message to be sent to server:\n");
    fgets(buffer, sizeof(buffer), stdin);

    // Send message
    if (sendto(sockfd,
               buffer,
               strlen(buffer),
               0,
               (struct sockaddr *)&server,
               sizeof(server)) < 0) {
        printf("Error in sendto()\n");
        return 0;
    }

    printf("Message sent successfully\n");

    close(sockfd);
    return 0;
}
⚠️ KEY DIFFERENCE FROM SERVER
Client uses argc != 3 (needs both IP and Port). inet_addr(argv[1]) converts dotted-decimal IP string → binary. Port is atoi(argv[2]). No bind(), no recvfrom().

Stop & Wait Protocol Flow Control

Send one packet → Wait for ACK → Send next. Simplest flow control protocol. Uses random error simulation with rand() % 4.

How It Works
1Sender sends packet with sequence number
2errorframe = rand() % 4 — simulates packet loss (0 = error)
3If no error: Receiver gets it, sends ACK back
4errorack = rand() % 4 — simulates ACK loss
5If ACK received: Sender moves to next packet
6If error: t-- (timeout countdown), retransmit
Key Variables
turn = 's' or 'r' (whose turn)
errorframe = 0 means error in packet
errorack = 0 means error in ACK
p.seq = sequence number
ak = acknowledgment number
frexp = expected frame number (receiver side)
The Core Logic
rand() % 4 == 0 means ~25% chance of error. This simulates real network unreliability. If errorframe ≠ 0, packet sent OK. If errorack ≠ 0, ACK sent OK.
Stop & Wait — Full Code
// STOP AND WAIT — Networks Lab
#include <stdio.h>
#include <stdlib.h>

struct frame {
    int info;
    int seq;
};

int ak;
int t = 5, k;
int disconnect = 0;
struct frame p;
char turn = 's';
int errorframe = 1, errorack = 1;

void sender();
void receiver();

void main() {
    p.info = 0;
    p.seq = 0;
    while (!disconnect) {
        sender();
        receiver();
    }
}

void sender() {
    static int flag = 0;
    if (turn == 's') {
        if (errorack == 0) {          // ACK was lost last time
            printf("SENDER: sent packet with seq NO:%d\n", p.seq);
            errorframe = rand() % 4;
            printf("%s\n", (errorframe == 0 ? "Error while sending Packet" : ""));
            turn = 'r';
        } else {
            if (flag == 1)
                printf("SENDER: Received ACK for packet %d\n", ak);
            if (p.seq == 5) {         // Sent all 5 packets
                disconnect = 1;
                return;
            }
            p.info = p.info + 1;
            p.seq = p.seq + 1;
            printf("SENDER: Sent packet with seq NO:%d\n", p.seq);
            errorframe = rand() % 4;
            printf("%s\n", (errorframe == 0 ? "Error While sending Packet" : ""));
            turn = 'r';
            flag = 1;
        }
    } else {
        t--;  // timeout countdown
    }
}

void receiver() {
    static int frexp = 1;  // expected frame
    if (turn == 'r') {
        if (errorframe != 0) {    // packet arrived OK
            if (p.seq == frexp) {  // correct sequence
                printf("RECEIVER: Received packet with seq %d\n", p.seq);
                ak = p.seq;
                frexp = frexp + 1;
                turn = 's';
                errorack = rand() % 4;
                printf("%s\n", (errorack == 0 ? "Error while sending ACK" : ""));
            } else {              // duplicate packet
                printf("RECEIVER: Duplicated packet with seq %d\n", p.seq);
                ak = frexp - 1;
                turn = 's';
                errorack = rand() % 4;
                printf("%s\n", (errorack == 0 ? "Error while sending ACK" : ""));
            }
        }
    }
}
Stop & Wait — Tricks to Remember
🔁 THE LOOP
While(!disconnect) calls sender() then receiver() alternately. They cooperate using the turn variable.
🎲 ERROR SIMULATION
rand() % 4 == 0 → 25% error chance. Non-zero = success. This simulates both frame errors AND ACK errors separately.
🔢 SEQUENCE LOGIC
Sender's p.seq increments each packet. Receiver's frexp (expected) also increments. If they match → new packet. If they don't → duplicate, re-ACK.
📝 EXAM Q: Why two error variables?
errorframe = error in the data packet going from sender to receiver. errorack = error in the ACK going back from receiver to sender. Both can fail independently.
Distance Vector Routing Routing Algorithm

Each router tells its neighbors what it knows. Based on Bellman-Ford algorithm. Each node maintains a routing table with distance and next-hop to every other node.

The Bellman-Ford Core Logic
THE KEY FORMULA
if (rt[i].dist[j] > dmat[i][k] + rt[k].dist[j])

Translation: "Can I reach j via intermediate node k cheaper than my current best path?" If yes → update!
1Initialize routing table from cost matrix (direct link costs)
2Triple nested loop: for every i (src) → j (dst) → k (intermediate)
3If shorter path found via k: update distance AND update "from" (next hop)
4Repeat until no updates (count == 0) → converged!
5Print final routing table for each node
DVR — Full Code
// DISTANCE VECTOR ROUTING — Networks Lab
#include <stdio.h>

struct node {
    unsigned dist[20];   // distance to each node
    unsigned from[20];   // next hop to reach each node
} rt[10];              // routing table for 10 nodes

int main() {
    int dmat[20][20];    // cost matrix
    int n, i, j, k, count = 0;

    printf("Enter the number of nodes: ");
    scanf("%d", &n);

    printf("Enter the cost matrix:\n");
    for (i = 0; i < n; i++)
        for (j = 0; j < n; j++)
            scanf("%d", &dmat[i][j]);

    // Initialize routing table with direct link costs
    for (i = 0; i < n; i++)
        for (j = 0; j < n; j++) {
            rt[i].dist[j] = dmat[i][j];
            rt[i].from[j] = j;  // initially, go directly
        }

    // Bellman-Ford: update until no change
    do {
        count = 0;
        for (i = 0; i < n; i++)
            for (j = 0; j < n; j++)
                for (k = 0; k < n; k++) {
                    // Can we reach j via k cheaper?
                    if (rt[i].dist[j] > dmat[i][k] + rt[k].dist[j]) {
                        rt[i].dist[j] = dmat[i][k] + rt[k].dist[j];  // ✅ FIXED
                        rt[i].from[j] = k;
                        count++;  // something changed
                    }
                }
    } while (count != 0);  // repeat till stable

    // Print routing tables
    for (i = 0; i < n; i++) {
        printf("\n\nState value for router %d\n", i + 1);
        for (j = 0; j < n; j++)
            printf("\n node %d via %d Distance %d",
                   j + 1, rt[i].from[j] + 1, rt[i].dist[j]);
    }
    printf("\n");
    return 0;
}
DVR — Tricks to Remember
🧠 THE MAGIC LINE
rt[i].dist[j] > dmat[i][k] + rt[k].dist[j]

Read as: "Current best distance from i→j" > "Cost of i→k (direct link)" + "k's best distance to j". If true → we found a shortcut via k!
🔄 WHY do-while?
We repeat until nothing changes (count == 0). This is convergence. Each iteration, more optimal paths propagate through the network. DVR is slow to converge — that's actually a known disadvantage!
📋 STRUCT TRICK
rt[i].dist[j] = shortest distance from node i to node j
rt[i].from[j] = which neighbor to use to get to j from i
💡 EXAM Q: What is the time complexity?
O(n³) per iteration × multiple iterations. Three nested loops over n nodes = O(n³). This is why DVR doesn't scale well to large networks.
Leaky Bucket Algorithm Congestion Control

Traffic shaping algorithm. Imagine a bucket with a hole — packets fill the bucket, but output rate is constant. Excess packets are dropped.

Visualization
Incoming packets (bursty)
→
BUCKET
size = buck_size
current = store
→
Outgoing (steady rate)
If incoming > (bucket_size - store) → DROP excess
Each round: store -= outgoing (constant drain)
Key Variables
buck_size = max bucket capacity
store = currently stored packets
incoming = new packets arriving
outgoing = constant drain rate
The Core Check
if (incoming <= buck_size - store)
→ fits in bucket → accept

else
→ drop excess, fill to max
Leaky Bucket — Full Code
// LEAKY BUCKET ALGORITHM — Networks Lab
#include <stdio.h>

int main() {
    int incoming, outgoing, buck_size, n, store = 0;

    printf("Enter bucket size, outgoing rate and no of inputs: ");
    scanf("%d%d%d", &buck_size, &outgoing, &n);

    while (n != 0) {
        printf("Enter the incoming packet size: ");
        scanf("%d", &incoming);
        printf("Incoming packet size %d\n", incoming);

        // Does it fit in bucket?
        if (incoming <= (buck_size - store)) {
            store += incoming;
            printf("Bucket buffer size %d out of %d\n", store, buck_size);
        } else {
            // Drop what doesn't fit
            printf("Dropped %d no of packets\n", incoming - (buck_size - store));
            printf("Bucket buffer size %d out of %d\n", buck_size, buck_size);
            store = buck_size;  // bucket is full
        }

        // Drain at constant outgoing rate
        store = store - outgoing;
        if (store < 0) store = 0;  // ✅ FIXED — clamp to zero

        printf("After outgoing %d packets left out of %d in buffer\n",
               store, buck_size);
        n--;
    }
    return 0;
}
Leaky Bucket — Tricks
💧 THE ANALOGY
Bucket = buffer. Water in = incoming packets. Hole = outgoing rate. If too much water → overflow (drop packets). Hole always drains at same rate regardless of input.
🔢 DROP CALCULATION
Dropped = incoming - (buck_size - store). This is: "how many packets exceeded available space?"
📝 EXAM Q: Leaky vs Token Bucket
Leaky = fixed output rate always (strict). Token Bucket = can burst occasionally (flexible). Leaky drops excess; Token stores tokens to allow bursts.
⚠️ EDGE CASE
After outgoing, if store goes negative — we clamp store to 0 (if (store < 0) store = 0;) to handle this.
File Transfer Protocol TCP-Based

FTP is TCP + file I/O. Server reads a file and sends its contents. Client receives it and writes to a new file. The socket setup is identical to TCP — only the data handling changes.

FTP Flow
1Server: socket → bind → listen → accept (same as TCP)
2Client: socket → connect (same as TCP)
3Client sends filename to server via send()
4Server opens that file with fopen(), reads line by line
5Server sends each line; Client receives and writes to new file
6Server sends "completed" when done; Client detects it
🔑 KEY INSIGHT
FTP = TCP socket code + File I/O (fopen, fgets, fputs). If you know TCP, you know 70% of FTP already!
FTP Server SERVER
// FTP SERVER — Networks Lab
#include <stdio.h>
#include <stdlib.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <string.h>

int main() {
    int sd, newsd, n, port;
    char name[100], fileread[100];
    FILE *fp;
    struct sockaddr_in serveraddr, clientaddr;
    socklen_t len = sizeof(clientaddr);

    printf("Enter the port: ");
    scanf("%d", &port);

    // TCP socket setup (same as TCP server!)
    sd = socket(AF_INET, SOCK_STREAM, 0);
    if (sd < 0)
        printf("Can't create socket\n");

    serveraddr.sin_family = AF_INET;
    serveraddr.sin_addr.s_addr = INADDR_ANY;
    serveraddr.sin_port = htons(port);

    if (bind(sd, (struct sockaddr*)&serveraddr, sizeof(serveraddr)) < 0)
        printf("Can't bind\n");

    listen(sd, 5);

    newsd = accept(sd, (struct sockaddr*)&clientaddr, &len);
    if (newsd < 0)
        printf("Can't accept\n");

    // Receive filename from client
    n = recv(newsd, name, 100, 0);
    name[n] = '\0';
    printf("File requested: %s\n", name);

    // Open the file
    fp = fopen(name, "r");
    if (fp == NULL) {
        send(newsd, "File not found", 100, 0);
    } else {
        // Send file line by line
        while (fgets(fileread, sizeof(fileread), fp)) {
            send(newsd, fileread, sizeof(fileread), 0);
            sleep(1);
        }
        // Signal completion
        send(newsd, "completed", 100, 0);
        fclose(fp);
    }

    close(newsd);
    close(sd);
    return 0;
}
FTP Client CLIENT
// FTP CLIENT — Networks Lab
#include <stdio.h>
#include <stdlib.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <string.h>

int main() {
    FILE *fp;
    int csd, port, n;
    char name[100], rcv[100], newname[100];
    struct sockaddr_in serveraddr;

    printf("Enter the port: ");
    scanf("%d", &port);

    // TCP socket setup (same as TCP client!)
    csd = socket(AF_INET, SOCK_STREAM, 0);
    if (csd < 0)
        printf("Can't create socket\n");

    serveraddr.sin_family = AF_INET;
    serveraddr.sin_addr.s_addr = INADDR_ANY;
    serveraddr.sin_port = htons(port);

    if (connect(csd, (struct sockaddr*)&serveraddr, sizeof(serveraddr)) < 0)
        printf("Can't connect\n");

    // Send requested filename to server
    printf("Enter filename to request: ");
    scanf("%s", name);
    send(csd, name, sizeof(name), 0);

    // Enter new filename to save as
    printf("Enter new filename to save: ");
    scanf("%s", newname);
    fp = fopen(newname, "w");

    // Receive and write until "completed"
    while ((n = recv(csd, rcv, 100, 0)) > 0) {
        if (strcmp(rcv, "completed") == 0) {
            printf("File transfer completed!\n");
            break;
        }
        fputs(rcv, fp);
    }

    fclose(fp);
    close(csd);
    return 0;
}
FTP — Copy-Paste Tricks
Server → Client Changes
✅ COPY ENTIRE TCP server/client boilerplate
Socket setup, bind, listen, accept (server) / connect (client) — 100% identical to TCP. Don't rewrite this!
➕ ADD: File handling
Server: fopen(name, "r") + fgets() loop + send("completed")
Client: fopen(newname, "w") + recv loop + strcmp("completed") check + fputs()
💡 TERMINATION TRICK
Server sends "completed" as a magic string to signal end of file. Client checks strcmp(rcv, "completed") == 0 to break the loop. This is the "EOF signal" over the network.
📋 FTP vs TCP diff in one sentence
FTP = TCP + fopen/fgets/fputs/fclose. Everything else is the same.

Q2 What is TCP/IP?
TCP & UDP ▶

TCP/IP stands for Transmission Control Protocol / Internet Protocol.

  • IP deals with packets — addressing and routing them across networks.
  • TCP enables two hosts to establish a connection and exchange streams of data.
  • TCP guarantees delivery and guarantees packets are delivered in the same order they were sent.
Q3 What is UDP?
TCP & UDP ▶

UDP (User Datagram Protocol) is an alternative to TCP used primarily for establishing low-latency and loss-tolerating connections between applications. It skips connection setup overhead — ideal for real-time apps like video streaming, DNS, gaming.

Q5 What is a Socket?
Socket Programming ▶

A socket is an endpoint for communication between two machines. It is identified by an IP address + port number. In C, socket() creates a socket descriptor used for all subsequent operations.

Q6 Explain the TCP server-client flow.
Socket Programming ▶

Server: socket() → bind() → listen() → accept() → recv()/send() → close()

Client: socket() → connect() → send()/recv() → close()

Key: Server uses TWO socket descriptors — one for listening (sock_desc), one returned by accept() for actual communication (temp_sock_desc).

Q8 What is AF_INET? What is SOCK_STREAM vs SOCK_DGRAM?
Socket Programming ▶
  • AF_INET — Address Family IPv4. Used to specify IPv4 protocol family.
  • SOCK_STREAM — TCP. Stream-based, reliable, connection-oriented.
  • SOCK_DGRAM — UDP. Datagram-based, unreliable, connectionless.
Q9 What does INADDR_ANY mean?
Socket Programming ▶

It tells the server socket to accept connections on any available network interface of the machine, rather than binding to a specific IP address.

Q10 What does htons() do?
Socket Programming ▶

htons() = Host TO Network Short. Converts a port number from host byte order to network byte order (big-endian). Networks always use big-endian; x86 machines are little-endian — so this conversion is necessary.

Q11 What does inet_addr() do?
Socket Programming ▶

Converts a dotted-decimal IP string (e.g., "127.0.0.1") into a binary network address (in_addr_t). Used in UDP clients that take an IP address as a command-line argument.

Q12 What does the client NOT need that the server does?
Socket Programming ▶

bind(), listen(), accept() — the client only needs socket() + connect(). The single connect() call replaces all three server-side calls.

Q13 Explain the UDP server-client flow.
Socket Programming ▶

Server: socket() → bind() → recvfrom() → close()

Client: socket() → sendto() → close()

No connect(), no accept(), no listen(). UDP is connectionless.

Q15 What is socklen_t and why must it be initialized before recvfrom()?
Socket Programming ▶

socklen_t holds the size of the sockaddr structure. recvfrom() takes a pointer to it as a value-result argument — it reads the size going in and writes the actual size coming out. If uninitialized, the kernel doesn't know how much space is available → undefined behavior / crash.

Q16 How does FTP differ from basic TCP?
Socket Programming ▶

FTP = TCP socket code + File I/O (fopen, fgets, fputs, fclose). The socket setup (bind, listen, accept / connect) is identical to TCP. The only additions are:

  • Server: fopen(filename, "r") → fgets() loop → send("completed")
  • Client: fopen(newname, "w") → recv() loop → strcmp("completed") check → fputs()
Q17 How does FTP signal end of file over the network?
Socket Programming ▶

The server sends the magic string "completed" after all file data is sent. The client checks strcmp(rcv, "completed") == 0 to break the receive loop. This acts as the EOF signal over the socket.

Q26 Define Network.
Fundamentals ▶

A network is a set of devices connected by physical media links. It is a connection of two or more nodes by a physical link, or two or more networks connected by nodes.

Q27 What is a Link?
Fundamentals ▶

The physical medium (e.g., coaxial cable, optical fiber) directly connecting two or more computers is called a Link.

Q28 What is a Node?
Fundamentals ▶

The computers/devices connected by links are called Nodes.

Q29 What is a Gateway / Router?
Fundamentals ▶

A node connected to two or more networks that forwards messages from one network to another. A gateway typically operates at upper OSI layers and translates between different protocols/formats. A router operates at Layer 3 and makes forwarding decisions based on IP addresses.

Q30 What is the difference between Gateway and Router?
Fundamentals ▶

A gateway operates at all 7 OSI layers and translates between completely different network architectures or data formats. A router operates at Layers 1–3 and routes packets between networks of the same protocol family.

Q31 What is Protocol?
Fundamentals ▶

A protocol is a set of rules that govern all aspects of information communication between devices.

Q32 What are the key elements of a Protocol?
Fundamentals ▶
  • Syntax — structure/format of data (order of presentation)
  • Semantics — meaning of each section of bits
  • Timing — when data should be sent and how fast
Q33 Define Bandwidth and Latency.
Fundamentals ▶
  • Bandwidth — number of bits that can be transmitted over the network in a given time period (measured in bps)
  • Latency — how long it takes a message to travel from one end of a network to the other (measured in time)
Q34 What is Round Trip Time (RTT)?
Fundamentals ▶

The time it takes to send a message from one end of a network to the other and back. RTT = 2 × one-way latency.

Q35 Define Unicasting, Multicasting, and Broadcasting.
Fundamentals ▶
  • Unicasting — message sent from one source to a single destination
  • Multicasting — message sent to a subset of nodes
  • Broadcasting — message sent to all nodes in the network
Q36 What is Routing?
Fundamentals ▶

The process of systematically determining how to forward messages toward destination nodes based on their address.

Q37 What is a subnet?
Fundamentals ▶

A subnet (subnetwork) is an identifiably separate part of an organization's network — typically all machines at one location or on the same LAN. Allows a single shared network address for internet connectivity.

Q38 What is semantic gap?
Fundamentals ▶

The gap between what applications expect from a network and what the underlying technology can actually provide is called the semantic gap.

Q39 What is Congestion?
Fundamentals ▶

When a switch receives packets faster than the shared link can handle and runs out of buffer space — some packets are dropped. This state is called congestion.

Q40 What are the criteria for an effective network?
Fundamentals ▶
  • Performance — transmit time, response time
  • Reliability — frequency of failure, recovery time
  • Security — protection from unauthorized access and viruses
Q41 What is Point-to-Point vs Multiple Access?
Fundamentals ▶
  • Point-to-Point — physical link limited to a pair of nodes
  • Multiple Access — physical link shared by more than two nodes
Q42 What are the two types of transmission technology?
Fundamentals ▶
  1. Broadcast — shared medium, all nodes receive
  2. Point-to-Point — dedicated link between two nodes
Q43 What is a peer-to-peer process?
Fundamentals ▶

Processes on each machine that communicate at the same layer of the OSI model are called peer-to-peer processes.

Q44 What is Encapsulation?
Fundamentals ▶

The practice of enclosing data from one protocol within the messages of another protocol. Each OSI layer adds its own header (and sometimes trailer) as data passes down the stack.

Q46 What is MTU?
Fundamentals ▶

Maximum Transmission Unit — the size (in bytes) of the largest protocol data unit a layer can pass onwards. Fixed by standards for Ethernet. Larger MTU = greater efficiency (less header overhead per byte of user data).

Q47 Why is fragmentation required?
Fundamentals ▶

Every network has an MTU. Packets larger than the MTU must be divided into smaller fragments to traverse that network. Each fragment is reassembled at the destination.

Q48 What is IPC? Name three techniques.
Fundamentals ▶

Inter-Process Communication (IPC) — a set of techniques for exchanging data among multiple threads/processes. Three techniques:

  1. Message passing
  2. Shared memory
  3. Remote Procedure Calls (RPC)
Q49 What are the two types of errors?
Error Control ▶
  • Single-bit error — only one bit in the data unit has changed
  • Burst error — two or more bits in the data unit have changed
Q50 What is Error Detection?
Error Control ▶

Uses redundancy (adding extra bits) to detect errors at the destination. Does not fix — only detects.

Q51 What are common Error Detection methods?
Error Control ▶
  • VRC — Vertical Redundancy Check (parity bit per data unit)
  • LRC — Longitudinal Redundancy Check (redundant row for a block)
  • CRC — Cyclic Redundancy Check (binary division, most powerful)
  • Checksum — used by higher-layer protocols (TCP/IP)
Q52 What is VRC?
Error Control ▶

A parity bit is added to every data unit so the total number of 1s becomes even (even parity). Detects all single-bit errors. Detects burst errors only if total errors in each data unit is odd.

Q53 What is LRC?
Error Control ▶

A block of bits is divided into rows and a redundant row is added to the whole block. Better than VRC for burst error detection, but can miss errors if two data units have the same bit positions damaged.

Q54 What is CRC?
Error Control ▶

Cyclic Redundancy Check — the most powerful redundancy checking technique. Based on binary division. A divisor polynomial is agreed upon; the remainder of dividing the data by the polynomial is appended. Receiver performs the same division — a non-zero remainder indicates an error.

Q55 What is Checksum?
Error Control ▶

Used by TCP/IP for error detection. Steps:

  1. Divide data into sections
  2. Add sections using 1's complement arithmetic
  3. Take the complement of the final sum → this is the checksum
Q56 Compare Error Detection vs Error Correction.
Error Control ▶
  • Detection — only checks if an error occurred (simpler)
  • Correction — must know the exact number and location of corrupted bits (much harder). Requires more redundant bits.
Q57 What is Forward Error Correction?
Error Control ▶

The receiver tries to guess the original message using redundant bits, without asking for retransmission. Used where retransmission is not feasible (e.g., satellite links).

Q58 What is Retransmission?
Error Control ▶

A technique where the receiver detects an error and asks the sender to resend the message. Repeated until an error-free copy is received.

Q59 What are other error detection/correction algorithms?
Error Control ▶

Parity check, LRC, CRC, Fire codes (FC), BCH (Bose-Chaudhuri-Hocquenghem), Reed-Solomon (RS), Hamming codes (HC), Turbo Codes, Viterbi Convolutional Coding (FEC), Golay.

Q60 What is Flow Control?
Flow Control ▶

A set of procedures used to restrict the amount of data a sender can send before waiting for acknowledgment. Prevents a fast sender from overwhelming a slow receiver.

Q61 What is Error Control?
Flow Control ▶

Combines error detection + correction. Allows the receiver to inform the sender of lost/damaged frames and coordinates retransmission. In the data link layer, implemented primarily as detection + retransmission.

Q62 What is ARQ (Automatic Repeat Request)?
Flow Control ▶

When an error is detected, specified frames are automatically retransmitted. This process is called ARQ. Error control in the data link layer is often implemented this way.

Q63 What is Stop-and-Wait Protocol?
Flow Control ▶

The sender sends one frame, waits for confirmation (ACK) from the receiver, then sends the next frame. Simplest flow control protocol. Inefficient for long-distance links (wastes bandwidth waiting).

Q64 What is Stop-and-Wait ARQ?
Flow Control ▶

Error correction in Stop-and-Wait where the sender keeps a copy of the sent frame and retransmits when the timer expires (ACK not received in time).

Q65 Why are sequence numbers used in reliable transmission?
Flow Control ▶

To number frames so the receiver can detect duplicates and out-of-order delivery. The sender and receiver use sequence numbers to agree on which frames have been received and which need retransmission.

Q66 What is Pipelining?
Flow Control ▶

A technique where a new task is begun before the previous one ends. In networking, the sender sends multiple frames without waiting for individual ACKs — improving link utilization.

Q67 What is a Sliding Window?
Flow Control ▶

An abstract concept defining the range of sequence numbers the sender/receiver must deal with at any time. Allows multiple outstanding (unacknowledged) frames. Window slides forward as ACKs are received.

Q68 What is Piggybacking?
Flow Control ▶

A technique where control information (ACKs) about received frames is carried along with outgoing data frames in the opposite direction, instead of sending a separate ACK. Improves efficiency.

Q69 Explain the Stop-and-Wait simulation — what do errorframe and errorack represent?
Flow Control ▶
  • errorframe — simulates error in the data packet going from sender to receiver. rand() % 4 == 0 (~25% chance) means the packet was corrupted.
  • errorack — simulates error in the ACK going back from receiver to sender. They fail independently — that's why two separate variables are needed.
Q70 What is the role of frexp in Stop-and-Wait?
Flow Control ▶

frexp (frame expected) is the receiver's expected sequence number. If p.seq == frexp, it's a new packet → accept and increment. If they don't match → duplicate packet → re-send ACK for last received frame.

Q71 What is Distance Vector Routing?
Routing & Addressing ▶

A routing algorithm where each router tells its neighbors what it knows about distances to all destinations. Based on the Bellman-Ford algorithm. Routers iteratively update their tables until convergence.

Q72 Explain the Bellman-Ford update formula used in DVR.
Routing & Addressing ▶ 
if (rt[i].dist[j] > dmat[i][k] + rt[k].dist[j])
    rt[i].dist[j] = dmat[i][k] + rt[k].dist[j]

Translation: "Can I reach destination j from node i via intermediate node k, cheaper than my current best path?" If yes → update distance and next hop.

Q73 What does rt[i].dist[j] and rt[i].from[j] store?
Routing & Addressing ▶
  • rt[i].dist[j] — shortest known distance from node i to node j
  • rt[i].from[j] — which neighbor (next hop) to use to get from i to j
Q74 Why does DVR use a do-while loop?
Routing & Addressing ▶

The algorithm repeats until no table entry changes (count == 0). Each iteration, better paths propagate through the network. This is called convergence. DVR is known to be slow to converge — a notable disadvantage.

Q75 What is the time complexity of DVR?
Routing & Addressing ▶

O(n³) per iteration × multiple iterations. Three nested loops (source × destination × intermediate) = O(n³). This is why DVR doesn't scale well to large networks.

Q76 What is the difference between ARP and RARP?
Routing & Addressing ▶
  • ARP (Address Resolution Protocol) — maps a 32-bit IP address → 48-bit MAC (physical) address. Sends a broadcast query; the target replies with its MAC.
  • RARP (Reverse ARP) — allows a host to discover its IP address when it knows only its physical (MAC) address.
Q77 What is ICMP?
Routing & Addressing ▶

Internet Control Message Protocol — a network layer protocol in TCP/IP. Used by hosts and gateways to send error/notification messages back to the sender (e.g., destination unreachable, time exceeded). Uses echo test/reply to check if a destination is reachable (ping).

Q78 What is RIP (Routing Information Protocol)?
Routing & Addressing ▶

A simple protocol used to exchange routing information between routers. Uses hop count as the metric. Maximum hop count = 15 (16 = infinity/unreachable). A type of Distance Vector protocol.

Q79 What is OSPF?
Routing & Addressing ▶

Open Shortest Path First — an internet routing protocol that scales well, can route traffic along multiple paths, and uses knowledge of the entire network topology (link-state) to make accurate routing decisions. Faster convergence than RIP.

Q80 What is BGP?
Routing & Addressing ▶

Border Gateway Protocol — used to advertise the set of networks reachable within an autonomous system. The standard exterior gateway protocol of the internet. Newer than EGP.

Q81 What is an Autonomous System?
Routing & Addressing ▶

A collection of routers under a single administrative authority using a common Interior Gateway Protocol (IGP).

Q82 What is IP address? What are the IP address classes?
Routing & Addressing ▶

An IP address is a numerical label assigned to devices on a network for identification and location addressing.

Q83 What is MAC address?
Routing & Addressing ▶

A unique identifier assigned to a network adapter/NIC by the manufacturer. Used at the MAC sublayer of the Data Link Layer. 48 bits, usually stored in ROM on the network card. Also called Ethernet Hardware Address (EHA) or physical address.

Q84 What is a Multi-homed Host?
Routing & Addressing ▶

A host that has multiple network interfaces and therefore requires multiple IP addresses.

Q85 What is the minimum and maximum header length in TCP/IP?
Routing & Addressing ▶

Both TCP segment and IP datagram headers: minimum 20 bytes, maximum 60 bytes.

Q86 What is Proxy ARP?
Routing & Addressing ▶

Using a router to answer ARP requests on behalf of a host. Done when the originating host believes a destination is local, but it actually lies beyond the router.

Q87 What is multicast routing?
Routing & Addressing ▶

Sending a message to a group of destinations is called multicasting. The routing algorithm used for this is called multicast routing.

Q88 What is Traffic Shaping?
Congestion Control ▶

A method to force packets to be transmitted at a more predictable, uniform rate to reduce congestion. Instead of bursty traffic, the network sees a smooth stream.

Q89 What is the Leaky Bucket Algorithm?
Congestion Control ▶

A traffic shaping algorithm. Think of a bucket with a hole:

  • Incoming packets fill the bucket (at any rate)
  • Output drains at a constant rate (the "leak")
  • If incoming > available space → excess packets are dropped
  • Result: smooth, constant output regardless of bursty input

Congestion Control ▶ 
Dropped = incoming - (buck_size - store)

i.e., how many packets exceeded the available space in the bucket.

Q93 What is bursty traffic?
Congestion Control ▶

An uneven pattern of data transmission — sometimes very high rate, sometimes very low. Real-world traffic (HTTP, video) is naturally bursty.

Q94 What is the silly window syndrome?
Congestion Control ▶

A TCP performance problem where data is passed in large blocks from the sender but an interactive application reads 1 byte at a time on the receiver side, causing tiny window sizes and extremely inefficient use of the network.

Q95 What is Multiplexing?
Multiplexing & Media ▶

A set of techniques that allows the simultaneous transmission of multiple signals across a single data link by sharing the link's capacity.

Q96 What are the categories of Multiplexing?

  • TDM — Time Division Multiplexing (digital)
    • Synchronous TDM
    • Asynchronous TDM (Statistical TDM)
  • WDM — Wave Division Multiplexing (optical fiber)

Multiplexing & Media ▶

The multiplexer allocates exactly the same time slot to each device at all times, whether or not the device has anything to transmit. Wastes slots but simple to implement.

Q100 What is Attenuation?
Multiplexing & Media ▶

The degeneration (weakening) of a signal over distance on a network cable. That's why repeaters are needed to regenerate signals over long distances.

Q101 Difference between Bit Rate and Baud Rate.
Multiplexing & Media ▶
  • Bit Rate — number of bits transmitted per second
  • Baud Rate — number of signal units per second required to represent those bits

Formula: Baud rate = Bit rate / N
where N = number of bits represented by each signal shift.

Q102 What is Bandwidth (physical definition)?
Multiplexing & Media ▶

Every transmission line has an upper and lower limit on the frequency of signals it can carry. This limited frequency range is called the bandwidth.

Q103 Explain the 5-4-3 Rule (Ethernet).
Multiplexing & Media ▶

Between any two points on an Ethernet network, there can be no more than 5 network segments, 4 repeaters, and of those 5 segments, only 3 can be populated with devices.

Q104 What is baseband vs broadband transmission?
Multiplexing & Media ▶
  • Baseband — entire cable bandwidth consumed by a single signal
  • Broadband — signals sent on multiple frequencies, allowing multiple simultaneous signals
Q105 What is Framing?
Security & Misc ▶

In the Data Link Layer, framing separates messages by adding sender and destination addresses. The destination address defines where to go; the sender address helps the recipient acknowledge receipt.

Q106 What is Bit Stuffing?
Security & Misc ▶

Adding an extra 0 whenever five consecutive 1s follow a 0 in the data, so the receiver doesn't mistake the bit pattern 0111110 for a flag (delimiter). Receiver removes the stuffed bit.

Q107 What is Character (Byte) Stuffing?
Security & Misc ▶

When a data section contains a byte with the same pattern as the flag, an escape character (ESC) is added before it. The receiver removes the ESC and treats the next byte as data, not a delimiter.

Q108 What is a Packet Filter?
Security & Misc ▶

A router with extra functionality that inspects every incoming/outgoing packet. Packets meeting defined criteria are forwarded; those that fail are dropped. Basic form of firewall.

Q109 What is Virtual Channel / Virtual Circuit?
Security & Misc ▶

A logical connection from one source to one destination (though multicast is possible). Also called a virtual channel. Provides the appearance of a dedicated path even over a shared network.

Q110 What is Virtual Path?
Security & Misc ▶

Along a transmission path from source to destination, a group of virtual circuits can be grouped together into a virtual path.

Q111 What is a Region (in hierarchical routing)?
Security & Misc ▶

In hierarchical routing, routers are divided into regions. Each router knows full detail about routing within its own region but nothing about the internal structure of other regions. Reduces routing table size.

Q112 What is Kerberos?
Security & Misc ▶

An authentication service developed at MIT. Uses encryption to prevent intruders from discovering passwords and gaining unauthorized access. Based on a trusted third-party key distribution center.

Q113 What is RAID?
Security & Misc ▶

Redundant Array of Independent Disks — a method for providing fault tolerance by using multiple hard disk drives. Data is distributed across drives for redundancy and/or performance.

Q114 What is Beaconing?
Security & Misc ▶

A process that allows a network to self-repair. Stations on the ring notify others when they are not receiving transmissions. Used in Token Ring and FDDI networks.

Q115 What is MAU (Token Ring)?
Security & Misc ▶

In Token Ring, the hub is called Multistation Access Unit (MAU).

Q116 What is Passive Topology?
Security & Misc ▶

When computers on the network simply listen and receive the signal without amplifying it. Linear bus is an example of passive topology.

Q117 What is a Brouter?
Security & Misc ▶

A hybrid device combining features of both a bridge and a router. Can route specific protocols and bridge others.

Q118 What is Frame Relay? Which layer?
Security & Misc ▶

Frame relay is a packet switching technology operating at the Data Link Layer (Layer 2). Designed for WANs; faster than older X.25 but with less error checking.

Q119 What is Terminal Emulation (Telnet)? Which layer?
Security & Misc ▶

Telnet is also called terminal emulation. It belongs to the Application Layer. Allows a user to log in and control a remote computer.

Q120 What is the difference between Simulation and Emulation?
Security & Misc ▶
  • Simulation — mimics the outward appearance/behavior of a system in software
  • Emulation — mimics the actual cause/process. Usually hardware-based; self-contained and can be connected to a development environment
Q121 What is Network Throughput?
Security & Misc ▶

Usually represented as an average and measured in bits per second (bps) or packets per second. An important indicator of network performance and quality. High packet loss → lower throughput.

Q122 What are key, ciphertext, and plaintext?
Security & Misc ▶
  • Plaintext — original readable information
  • Ciphertext — result of encrypting plaintext using an algorithm (cipher) + key
  • Key — special knowledge used to encrypt/decrypt; without it the ciphertext is unreadable
  • Decryption — process of converting ciphertext back to plaintext
Q123 What is a Linear Block Code?
Security & Misc ▶

A block code where the XOR (modulo-2 addition) of any two valid codewords produces another valid codeword.

Q124 What are Cyclic Codes?
Security & Misc ▶

Special linear block codes with an extra property: if a codeword is cyclically shifted (rotated), the result is another valid codeword.

Q125 What is Redundancy (in error detection)?
Security & Misc ▶

Including extra bits in transmission solely for comparison at the destination. These bits add no information — their purpose is purely to enable error detection.
