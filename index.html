TCP — Transmission Control Protocol Connection-Based

TCP is like a phone call — you establish a connection first, then talk, then hang up. Reliable, ordered delivery guaranteed.

How TCP Works — The Flow
1socket() — Both server & client create a socket (endpoint)
↓ Server only
2bind() — Server attaches socket to a port (3003)
3listen() — Server starts waiting for connections
4accept() — Server accepts incoming client
↓ Client only
5connect() — Client connects to server IP:port
↓ Both
6send() / recv() — Exchange data
7close() — Terminate connection
SERVER
socket() → bind() → listen() → accept()
↕3-Way Handshake (SYN, SYN-ACK, ACK)
CLIENT
socket() → connect()
↕send() / recv() both directions
Key Socket Args
AF_INET
IPv4 address family
SOCK_STREAM
TCP — stream based, reliable
INADDR_ANY
Accept connections on any interface
TCP Server SERVER

Server binds to port, listens, accepts client, then does recv → send.

#include <stdio.h>
#include <stdlib.h>
#include <netinet/in.h>
#include <unistd.h>
#include <string.h>
#include <sys/socket.h>

// Typedef for socket address structure
typedef struct sockaddr SA;

void main(){

    // sockfd   -> Server socket descriptor
    // clientfd -> Client socket descriptor
    int sockfd, clientfd, len, k;

    // Structures for server and client address
    struct sockaddr_in server, client;

    // Buffer for communication
    char buff[100];

    // Create TCP socket
    sockfd = socket(AF_INET, SOCK_STREAM, 0);

    if(sockfd == -1)
        printf("Error while SOCKET Creation\n");

    // Configure server details
    server.sin_addr.s_addr = INADDR_ANY;
    server.sin_family = AF_INET;
    server.sin_port = htons(3003);

    // Bind socket to IP and port
    k = bind(sockfd, (SA*)&server, sizeof(server));

    if(k == -1)
        printf("Error while Binding\n");

    // Put server in listening state
    k = listen(sockfd, 5);

    if(k == -1)
        printf("Error while Listening\n");

    len = sizeof(client);

    // Accept incoming client connection
    clientfd = accept(sockfd, (SA*)&client, &len);

    if(clientfd == -1)
        printf("Error while Accepting\n");

    // Receive message from client
    k = recv(clientfd, buff, 100, 0);

    printf("Message from Client is:\n\t\t%s\n\n", buff);

    // Send response to client
    printf("Enter Data to be Sent:\t");
    fgets(buff, 100, stdin);

    k = send(clientfd, buff, 100, 0);

    // Close sockets
    close(clientfd);
    close(sockfd);
}
🔑 REMEMBER
Server uses TWO sockets: sock_desc (listening) and temp_sock_desc (actual communication with client). accept() returns the second one.
TCP Client CLIENT

Client just creates socket, connects, then does send → recv. No bind/listen/accept needed!

#include <stdio.h>
#include <stdlib.h>
#include <netinet/in.h>
#include <unistd.h>
#include <string.h>
#include <sys/socket.h>

// Typedef for socket address structure
typedef struct sockaddr SA;

void main(){

    // clientfd -> Client socket descriptor
    // len,k    -> Helper variables
    int clientfd, len, k;

    // Structure to hold client/server address
    struct sockaddr_in client;

    // Buffer for message transmission
    char buff[100];

    // Create TCP socket
    clientfd = socket(AF_INET, SOCK_STREAM, 0);

    if(clientfd == -1)
        printf("Error while SOCKET Creation\n");

    // Configure server address details
    client.sin_addr.s_addr = INADDR_ANY;
    client.sin_family = AF_INET;
    client.sin_port = htons(3003);

    // Connect to server
    k = connect(clientfd, (SA*)&client, sizeof(client));

    if(k == -1)
        printf("Error while Connecting\n");

    // Read message from user
    printf("Enter Data to be Sent:\t");
    fgets(buff, 100, stdin);

    // Send data to server
    k = send(clientfd, buff, 100, 0);

    // Receive response from server
    k = recv(clientfd, buff, 100, 0);

    printf("Message from Server is:\n\t\t%s\n\n", buff);

    // Close socket connection
    close(clientfd);
}
TCP — Algorithm 30 Marks
30
Algorithm covers: Purpose · Data Structures · Methods · Pseudocode (Server + Client) · Step-by-step explanation
Purpose
TCP (Transmission Control Protocol) provides reliable, ordered, connection-oriented byte-stream communication between two processes over a network. It uses a three-way handshake to establish a connection before any data transfer begins, ensuring both endpoints are ready.
Data Structures & Variables
  • sockfdint — Server's listening socket descriptor returned by socket().
  • clientfdint — Descriptor for the accepted client connection, returned by accept().
  • sockaddr_instruct — Holds IP family, IP address (sin_addr), and port (sin_port) in network byte order.
  • buff[100]char array — Buffer used to hold data being sent or received over the socket.
  • SAtypedef — Alias for struct sockaddr; used for casting in bind/connect/accept calls.
System Call Methods
  • socket()socket(AF_INET, SOCK_STREAM, 0) — Creates a TCP socket. AF_INET = IPv4. SOCK_STREAM = reliable stream (TCP). Returns a file descriptor.
  • bind()bind(sockfd, &server, size) — Attaches the socket to an IP address and port number on the host machine. Only the server calls this.
  • listen()listen(sockfd, backlog) — Marks the socket as passive (ready to accept). backlog=5 = max queue of pending connections.
  • accept()accept(sockfd, &client, &len) — Blocks until a client connects; returns a new socket descriptor for that specific client.
  • connect()connect(fd, &server, size) — Client initiates the 3-way handshake to the server's IP:port. Blocks until connected.
  • send()send(fd, buff, len, 0) — Sends data through the connected socket. Returns number of bytes actually sent.
  • recv()recv(fd, buff, len, 0) — Receives data from the connected socket. Blocks until data arrives.
  • close()close(fd) — Closes the socket and releases the file descriptor. Server closes both clientfd and sockfd.
Pseudocode — TCP Server
BEGIN TCP_Server // Step 1: Create a TCP socket endpoint sockfd ← socket(AF_INET, SOCK_STREAM, 0) IF sockfd == -1 THEN print "Socket error" // Step 2: Configure server address server.sin_family ← AF_INET server.sin_addr ← INADDR_ANY // Accept on any interface server.sin_port ← htons(3003) // Convert port to network byte order // Step 3: Bind socket to address k ← bind(sockfd, &server, sizeof(server)) IF k == -1 THEN print "Bind error" // Step 4: Start listening for connections k ← listen(sockfd, 5) IF k == -1 THEN print "Listen error" // Step 5: Accept incoming client connection (BLOCKS here) clientfd ← accept(sockfd, &client, &len) IF clientfd == -1 THEN print "Accept error" // Step 6: Receive message from client recv(clientfd, buff, 100, 0) print "Message from Client: " + buff // Step 7: Read reply from keyboard and send fgets(buff, stdin) send(clientfd, buff, 100, 0) // Step 8: Close both sockets close(clientfd) close(sockfd) END TCP_Server
  • Step 1socket() — Creates the endpoint. SOCK_STREAM requests TCP. Returns an integer file descriptor used in all subsequent calls.
  • Step 2Address config — INADDR_ANY means "accept connections on any network interface on this machine." htons() converts from host to network byte order (big-endian).
  • Step 3bind() — Pairs the socket with the address struct. Without this, the OS doesn't know which port to listen on.
  • Step 4listen() — Moves socket to passive mode. Backlog 5 = at most 5 connections can queue while one is being served.
  • Step 5accept() — Creates a NEW socket dedicated to this client. The original sockfd keeps listening. This is the 3-way handshake completion point.
  • Steps 6–7recv() then send() — Server receives first, then replies. Order matters — client does the opposite (send first, recv second).
  • Step 8close() — Sends FIN to client, completing 4-way connection teardown. Both descriptors must be closed.
Pseudocode — TCP Client
BEGIN TCP_Client // Step 1: Create TCP socket clientfd ← socket(AF_INET, SOCK_STREAM, 0) IF clientfd == -1 THEN print "Socket error" // Step 2: Configure server address (same port as server) client.sin_family ← AF_INET client.sin_addr ← INADDR_ANY client.sin_port ← htons(3003) // Step 3: Connect to server (triggers 3-way handshake) k ← connect(clientfd, &client, sizeof(client)) IF k == -1 THEN print "Connect error" // Step 4: Read data from keyboard and send to server fgets(buff, stdin) send(clientfd, buff, 100, 0) // Step 5: Receive server's reply recv(clientfd, buff, 100, 0) print "Message from Server: " + buff // Step 6: Close socket close(clientfd) END TCP_Client
  • Step 1socket() — Same as server. Client uses only one socket descriptor (no separate listening socket needed).
  • Step 2Address config — Client configures the server's address to connect to, not its own. Port must match the server's bound port.
  • Step 3connect() — This single call replaces bind+listen+accept. It sends SYN, waits for SYN-ACK, then sends ACK. Blocks until the 3-way handshake completes.
  • Steps 4–5send() then recv() — Client sends first, then waits for server's reply. Reverse of server order.
📝 30-MARK EXAM SUMMARY
TCP algorithm = Socket Creation → Address Config → bind (S) / connect (C) → listen+accept (S only) → send/recv (opposite order on each side) → close. Key: server uses 2 sockets, client uses 1. htons() is mandatory for port. accept() is where the 3-way handshake finalises.

UDP — User Datagram Protocol Connectionless

UDP is like sending a letter — no handshake, no guarantee. Faster but unreliable. This version uses command-line args for IP/port and is one-way: client sends, server only receives. Key headers: <arpa/inet.h> (client) and <unistd.h> (server).

UDP Server SERVER

Run as: ./server <Port>. Server binds to the given port, prints Server waiting..., then blocks on recvfrom(). When a datagram arrives it prints it and exits. No reply sent — one-way only.

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/socket.h>
#include <unistd.h>
#include <netinet/in.h>

// Typedef for socket address
typedef struct sockaddr SA;

int main(int argc, char *argv[]){

    // Structures for server and client address
    struct sockaddr_in server, client;

    // sockfd -> Socket descriptor
    int sockfd, k;

    // Validate command line arguments
    if(argc != 2){

        printf("USAGE: %s <Port>\n", argv[0]);
        return 0;
    }

    // Create UDP socket
    sockfd = socket(AF_INET, SOCK_DGRAM, 0);

    if(sockfd == -1)
        printf("Error While Socket Creation\n");

    // Configure server details
    server.sin_addr.s_addr = INADDR_ANY;
    server.sin_port = htons(atoi(argv[1]));
    server.sin_family = AF_INET;

    // Bind socket to given port
    k = bind(sockfd, (SA*)&server, sizeof(server));

    if(k == -1)
        printf("Error while binding\n");

    socklen_t len = sizeof(client);

    // Buffer for communication
    char buff[100];

    printf("\t\tSERVER UP AND RUNNING\n\n");

    // Receive datagram from client
    k = recvfrom(sockfd, buff, 100, 0,
                 (SA*)&client, &len);

    if(k == -1)
        printf("Error while Receiving\n");

    printf("DATA GRAM RECEIVED:\n\t\t%s\n", buff);

    // Send response back to client
    printf("Enter the data to send\n");
    fgets(buff, 100, stdin);

    k = sendto(sockfd, buff, strlen(buff), 0,
               (SA*)&client, sizeof(client));
}
🔑 REMEMBER
Server uses argc != 2 — only takes port as arg. atoi(argv[1]) converts the port string to int. socklen_t client_len must be initialised with sizeof(client) before passing its address to recvfrom().
UDP Client CLIENT

Run as: ./client <IP Address> <Port>. Client takes the server IP and port on the command line, uses inet_addr() to convert the IP string, sends one message, then exits. No recvfrom() — one-way only.

UDP — Algorithm 30 Marks
30
Algorithm covers: Purpose · Data Structures · Methods · Pseudocode (Server + Client) · Step-by-step explanation
Purpose
UDP (User Datagram Protocol) provides a connectionless, unreliable, low-overhead datagram service. There is no handshake — the sender fires packets without confirming the receiver is ready. Used where speed > reliability (DNS, video streaming, gaming). This implementation takes port/IP as command-line arguments and performs a two-way exchange.
Data Structures & Variables
  • sockfdint — UDP socket descriptor created with SOCK_DGRAM.
  • sockaddr_in server/clientstruct — Holds address info. Server uses both (client address captured in recvfrom). Client uses only server struct.
  • socklen_t lenunsigned int — Holds the size of the address struct. Passed by pointer to recvfrom() so the kernel can fill the sender's address.
  • argv[]char* — Command-line args. Server: argv[1]=Port. Client: argv[1]=IP, argv[2]=Port.
  • buff[100]char array — Buffer for datagram data.
System Call Methods
  • socket()socket(AF_INET, SOCK_DGRAM, 0) — Creates a UDP socket. SOCK_DGRAM = datagram mode. No connection, no guarantee.
  • bind()bind(sockfd, &server, size) — Server only. Tells OS which port to receive datagrams on. Client does NOT bind — OS assigns ephemeral port automatically.
  • recvfrom()recvfrom(fd, buff, len, 0, &client, &clen) — Receives a datagram AND captures the sender's address into the client struct (needed to reply).
  • sendto()sendto(fd, buff, len, 0, &dest, dlen) — Sends datagram to the specified address. No connection needed — destination specified per-call.
  • atoi()atoi(argv[n]) — Converts command-line port string to integer before passing to htons().
Pseudocode — UDP Server
BEGIN UDP_Server(argv[1] = Port) IF argc ≠ 2 THEN print "USAGE: ./server <Port>" ; exit // Step 1: Create UDP socket sockfd ← socket(AF_INET, SOCK_DGRAM, 0) IF sockfd == -1 THEN print "Socket error" // Step 2: Configure and bind server address server.sin_family ← AF_INET server.sin_addr ← INADDR_ANY server.sin_port ← htons(atoi(argv[1])) bind(sockfd, &server, sizeof(server)) // Step 3: Wait for and receive a datagram print "SERVER UP AND RUNNING" len ← sizeof(client) recvfrom(sockfd, buff, 100, 0, &client, &len) // client struct now holds the sender's IP and port print "DATAGRAM RECEIVED: " + buff // Step 4: Read reply and send back to client fgets(buff, stdin) sendto(sockfd, buff, strlen(buff), 0, &client, sizeof(client)) END UDP_Server
  • argc checkServer expects exactly 2 args (program name + port). atoi(argv[1]) converts the port string to integer. Without this check the program would crash on wrong usage.
  • SOCK_DGRAMThis is the key difference from TCP. Datagram sockets are message-oriented — each sendto/recvfrom is one complete message, not a stream.
  • recvfrom & clientThe &client and &len parameters are filled by the kernel with the sender's address. This is how the server knows who to reply to — there is no established connection to reference.
  • sendto with clientUses the client struct populated by recvfrom as the destination. The server could not reply without this.
Pseudocode — UDP Client
BEGIN UDP_Client(argv[1] = IP, argv[2] = Port) IF argc ≠ 3 THEN print "USAGE: ./client <IP> <Port>" ; exit // Step 1: Create UDP socket sockfd ← socket(AF_INET, SOCK_DGRAM, 0) // Step 2: Configure server address server.sin_family ← AF_INET server.sin_addr ← INADDR_ANY // or inet_addr(argv[1]) server.sin_port ← htons(atoi(argv[2])) // Step 3: Read data and send datagram to server fgets(buff, stdin) sendto(sockfd, buff, strlen(buff), 0, &server, sizeof(server)) // Step 4: Receive server's reply len ← sizeof(server) recvfrom(sockfd, buff, 100, 0, &server, &len) print "DATAGRAM RECEIVED: " + buff END UDP_Client
  • No bind()Client does NOT bind. The OS automatically assigns a free ephemeral port when sendto() is first called. This is a fundamental UDP characteristic.
  • No connect()UDP client does not call connect(). Each sendto() specifies the destination explicitly — no persistent connection exists.
  • strlen(buff)UDP uses strlen(buff) as the length, not a fixed 100. This sends only the actual data, not the entire buffer — important for datagram efficiency.
  • argc != 3Client takes two args: IP address and port. It needs to know the server's address since there is no pre-established connection.

Stop & Wait Protocol Flow Control

Send one packet → Wait for ACK → Send next. Simplest flow control protocol. Uses random error simulation with rand() % 4.

How It Works
1Sender sends packet with sequence number
2errorframe = rand() % 4 — simulates packet loss (0 = error)
3If no error: Receiver gets it, sends ACK back
4errorack = rand() % 4 — simulates ACK loss
5If ACK received: Sender moves to next packet
6If error: t-- (timeout countdown), retransmit
Key Variables
turn = 's' or 'r' (whose turn)
errorframe = 0 means error in packet
errorack = 0 means error in ACK
p.seq = sequence number
ak = acknowledgment number
frexp = expected frame number (receiver side)
The Core Logic
rand() % 4 == 0 means ~25% chance of error. This simulates real network unreliability. If errorframe ≠ 0, packet sent OK. If errorack ≠ 0, ACK sent OK.
Stop & Wait — Full Code
#include <stdio.h>
#include <stdlib.h>
#include <stdbool.h>

// Frame structure used for transmission
struct frame{
    int seq, info;
};

// Global variables
int ack, errorFrame = 0, errorAck = 0, t = 5;
char turn = 's';
struct frame f;
bool disconnected = false;


// Sender function
// Handles packet transmission and retransmission
void sender(){

    static int flag = 0;

    if(turn == 's'){

        // If ACK was lost previously,
        // resend same packet
        if(errorAck == 0){

            printf("SENDER: ACK NOT RECEIVED< RESENDING PACKET WITH SEQ NO>: %d\n",
                   f.seq);

            // Randomly simulate frame error
            errorFrame = rand() % 4;

            if(errorFrame == 0){
                printf("Sender: Error While Sending Packet %d",
                       f.seq);
            }

            turn = 'r';
        }

        else{

            // Display ACK received message
            if(flag == 1){

                printf("SENDER: RECEIVED ACK FROM RECEIVER FOR PACKET %d\n",
                       ack);
            }

            // Stop after sending 5 packets
            if(f.seq == 5){
                disconnected = true;
                return;
            }

            // Generate next frame
            f.seq++;
            f.info++;

            printf("SENDER: SENDING PACKET WITH SEQ NO: %d\n",
                   f.seq);

            // Randomly simulate frame corruption
            errorFrame = rand() % 4;

            if(errorFrame == 0){

                printf("SENDER: ERROR WHILE SENDING PACKET WITH SEQ NO: %d\n",
                       f.seq);
            }

            turn = 'r';
            flag = 1;
        }
    }

    else{

        // Timer simulation
        if(t > 0)
            t--;
    }
}


// Receiver function
// Handles receiving frames and sending ACKs
void receiver(){

    static int exp = 1;

    if(turn == 'r'){

        // Proceed only if frame is not corrupted
        if(errorFrame != 0){

            // Correct frame received
            if(f.seq == exp){

                printf("RECEIVER: RECEIVED PACKET WITH SEQ NO: %d\n",
                       f.seq);

                ack = f.seq;

                // Switch turn back to sender
                turn = 's';

                exp++;

                // Randomly simulate ACK loss
                errorAck = rand() % 4;

                if(errorAck == 0)
                    printf("RECIEVER: ERROR WHILE SENDING ACK");
            }

            else{

                // Duplicate frame received
                printf("RECEIVER: RECEIVED DUPLICATE PACKET %d\n",
                       f.seq);

                ack = exp - 1;

                turn = 's';

                errorAck = rand() % 4;

                if(errorAck == 0)
                    printf("RECIEVER: ERROR WHILE SENDING ACK");
            }
        }
    }
}


// Main function starts sender and receiver simulation
void main(){

    // Initial frame setup
    f.seq = 0;
    f.info = 0;

    // Continue until all packets transmitted
    while(!disconnected){

        sender();
        receiver();
    }
}
Stop & Wait ARQ — Algorithm 30 Marks
30
Algorithm covers: Purpose · Data Structures · Sender Logic · Receiver Logic · Full Pseudocode · Step-by-step explanation
Purpose
Stop-and-Wait ARQ (Automatic Repeat reQuest) is the simplest error control and flow control protocol. The sender transmits one frame and then stops and waits for an acknowledgment (ACK) before sending the next. If the ACK is not received (due to frame error or ACK error), the sender retransmits. This implementation simulates errors using rand() % 4 == 0 (~25% error rate).
Data Structures & Variables
  • struct frame { seq, info }The data unit being transmitted. seq = sequence number (1–5). info = payload data.
  • turn (char)'s' = sender's turn, 'r' = receiver's turn. Controls who acts in each simulation cycle.
  • errorFrame (int)0 = frame corrupted during transmission (packet lost). Non-zero = frame delivered OK.
  • errorAck (int)0 = ACK lost on its way back to sender. Non-zero = ACK delivered OK.
  • ack (int)Acknowledgment number sent by receiver = sequence number of the successfully received frame.
  • disconnected (bool)Set to true when all 5 frames are transmitted successfully. Terminates the simulation loop.
  • t (int)Timeout counter (starts at 5). Decremented while waiting; triggers retransmission.
Methods
  • sender()Called each cycle when turn == 's'. Checks if previous ACK arrived. If errorAck == 0, retransmits. Otherwise, sends the next frame and randomizes a new errorFrame.
  • receiver()Called each cycle when turn == 'r'. If errorFrame != 0 (frame OK), checks if sequence matches expected. Sends ACK and randomizes errorAck.
  • rand() % 4Generates 0,1,2,3 with equal probability. 0 = error (~25% chance). This simulates real-world packet loss independently for frames and ACKs.
  • main()Initialises frame to seq=0, info=0. Runs while(!disconnected) loop calling sender() then receiver() alternately.
Pseudocode — Sender
FUNCTION sender(): IF turn == 's': IF errorAck == 0: // ACK was lost → retransmit print "ACK NOT RECEIVED, RESENDING SEQ: " + f.seq errorFrame ← rand() % 4 // Simulate new send attempt IF errorFrame == 0: print "Error sending packet" turn ← 'r' ELSE: // ACK OK → send next frame IF flag == 1: print "ACK RECEIVED FOR PACKET: " + ack IF f.seq == 5: // All 5 frames done disconnected ← TRUE RETURN f.seq ← f.seq + 1 f.info ← f.info + 1 print "SENDING PACKET SEQ: " + f.seq errorFrame ← rand() % 4 IF errorFrame == 0: print "Error sending packet" turn ← 'r' flag ← 1 ELSE: // Waiting for receiver → decrement timer IF t > 0: t ← t - 1
Pseudocode — Receiver
FUNCTION receiver(): IF turn == 'r': IF errorFrame ≠ 0: // Frame arrived without error IF f.seq == exp: // Expected frame — accept it print "RECEIVED PACKET SEQ: " + f.seq ack ← f.seq turn ← 's' exp ← exp + 1 errorAck ← rand() % 4 IF errorAck == 0: print "ACK send error" ELSE: // Duplicate frame — re-ACK old print "RECEIVED DUPLICATE PACKET: " + f.seq ack ← exp - 1 turn ← 's' errorAck ← rand() % 4 // If errorFrame == 0: frame corrupted → silently drop, no ACK sent FUNCTION main(): f.seq ← 0 ; f.info ← 0 WHILE NOT disconnected: sender() receiver()
  • errorAck == 0Retransmit path — ACK was lost so sender never got confirmation. It resends the same frame (f.seq unchanged). This is the ARQ "repeat" mechanism.
  • f.seq == 5Termination — After sending and getting ACK for all 5 packets, disconnected = true exits the while loop cleanly.
  • errorFrame != 0Frame OK check — Only when errorFrame is non-zero does the receiver process the frame. If it's 0, the frame is silently dropped, mimicking a corrupted/lost packet.
  • f.seq == expDuplicate detection — If seq doesn't match expected, it's a retransmitted duplicate. Receiver re-ACKs with ack = exp - 1 to tell sender "I already have this one."
  • static expThe static int exp = 1 in receiver() persists across calls — it remembers the next expected frame number across the entire simulation.
📝 30-MARK EXAM SUMMARY
Stop & Wait: sender sends 1 frame → waits for ACK. Two independent errors: errorFrame (packet lost) and errorAck (ACK lost). rand()%4==0 = ~25% error. Receiver drops silently on frame error; sender retransmits if ACK error. turn variable alternates control. Terminates after 5 successful frames.
Stop & Wait — Tricks to Remember
🔁 THE LOOP
While(!disconnect) calls sender() then receiver() alternately. They cooperate using the turn variable.
🎲 ERROR SIMULATION
rand() % 4 == 0 → 25% error chance. Non-zero = success. This simulates both frame errors AND ACK errors separately.
🔢 SEQUENCE LOGIC
Sender's p.seq increments each packet. Receiver's frexp (expected) also increments. If they match → new packet. If they don't → duplicate, re-ACK.
📝 EXAM Q: Why two error variables?
errorframe = error in the data packet going from sender to receiver. errorack = error in the ACK going back from receiver to sender. Both can fail independently.
Distance Vector Routing Routing Algorithm

Each router tells its neighbors what it knows. Based on Bellman-Ford algorithm. Each node maintains a routing table with distance and next-hop to every other node.

The Bellman-Ford Core Logic
THE KEY FORMULA
if (rt[i].dist[j] > dmat[i][k] + rt[k].dist[j])

Translation: "Can I reach j via intermediate node k cheaper than my current best path?" If yes → update!
1Initialize routing table from cost matrix (direct link costs)
2Triple nested loop: for every i (src) → j (dst) → k (intermediate)
3If shorter path found via k: update distance AND update "from" (next hop)
4Repeat until no updates (count == 0) → converged!
5Print final routing table for each node
DVR — Full Code
#include <stdio.h>

// Structure representing routing table for each router
struct node{
    unsigned dist[20];   // Minimum distance to each node
    unsigned from[20];   // Next hop router
}rt[10];

int main(){

    // dmat  -> Cost adjacency matrix
    // n     -> Number of routers
    // i,j,k -> Loop variables
    // count -> Tracks updates in routing table
    int dmat[20][20], n, i, j, k, count = 0;

    // Input number of routers
    printf("Enter the NO of node:\t");
    scanf("%d", &n);

    // Input cost matrix
    printf("Enter the Cost Matrix\n");
    for(i = 0; i < n; i++){
        for(j = 0; j < n; j++){
            scanf("%d", &dmat[i][j]);
        }
    }

    // Initialize routing table
    // Initially every router knows only direct distances
    for(i = 0; i < n; i++){
        for(j = 0; j < n; j++){

            // Direct distance from router i to j
            rt[i].dist[j] = dmat[i][j];

            // Direct router itself is next hop initially
            rt[i].from[j] = j;
        }
    }

    // Apply Distance Vector Routing Algorithm
    // Repeat until no routing table changes occur
    do{
        count = 0;

        for(i = 0; i < n; i++){
            for(j = 0; j < n; j++){
                for(k = 0; k < n; k++){

                    // Check if route through router k is shorter
                    if(rt[i].dist[j] >
                       dmat[i][k] + rt[k].dist[j]){

                        // Update shortest distance
                        rt[i].dist[j] =
                        dmat[i][k] + rt[k].dist[j];

                        // Update next hop
                        rt[i].from[j] = k;

                        count++;
                    }
                }
            }
        }

    }while(count != 0);

    // Display final routing tables
    for(i = 0; i < n; i++){

        printf("\n\nState Value for Router %d\n", i + 1);

        for(j = 0; j < n; j++){

            printf("\nnode %d via %d cost: %d",
                   j + 1,
                   rt[i].from[j] + 1,
                   rt[i].dist[j]);
        }

        printf("\n");
    }
}

Purpose
Distance Vector Routing is a dynamic routing algorithm based on the Bellman-Ford equation. Each router maintains a table of the minimum cost to reach every other router, along with the next-hop router to use. Routers share their tables with neighbors and iteratively update until no further improvement is possible — this is called convergence.
Data Structures & Variables
  • struct node { dist[20], from[20] } rt[10]Routing table array. rt[i].dist[j] = shortest known distance from router i to j. rt[i].from[j] = which next-hop router to use from i to reach j.
  • dmat[20][20]Direct cost adjacency matrix. dmat[i][j] = cost of direct link between i and j. 0 if same node, large number (999/INF) if no direct link.
  • nNumber of routers in the network.
  • i, j, kLoop indices. i = source router, j = destination router, k = intermediate (relay) router being evaluated.
  • countCounts updates made in each iteration. When count == 0, no more improvements possible → algorithm has converged.
The Bellman-Ford Core Condition
THE KEY FORMULA
if ( rt[i].dist[j] > dmat[i][k] + rt[k].dist[j] )

Plain English: "Is the cost to go from router i to router j via intermediate router k cheaper than my current best known route to j?"

If YES → Update: rt[i].dist[j] = dmat[i][k] + rt[k].dist[j] and rt[i].from[j] = k
Full Pseudocode
BEGIN DistanceVectorRouting // Step 1: Input READ n // number of routers READ dmat[n][n] // direct cost matrix // Step 2: Initialize routing tables FOR i ← 0 to n-1: FOR j ← 0 to n-1: rt[i].dist[j] ← dmat[i][j] // start with direct links only rt[i].from[j] ← j // initially go direct to j // Step 3: Iteratively apply Bellman-Ford REPEAT: count ← 0 FOR i ← 0 to n-1: // for each source router FOR j ← 0 to n-1: // for each destination FOR k ← 0 to n-1: // try every intermediate router IF rt[i].dist[j] > dmat[i][k] + rt[k].dist[j]: rt[i].dist[j] ← dmat[i][k] + rt[k].dist[j] rt[i].from[j] ← k // route via k is better count ← count + 1 UNTIL count == 0 // converged: no more updates // Step 4: Print routing tables FOR i ← 0 to n-1: print "Router " + (i+1) FOR j ← 0 to n-1: print "To node " + (j+1) + " via " + (rt[i].from[j]+1) + " cost: " + rt[i].dist[j] END DistanceVectorRouting
  • Step 1Input — The cost matrix is provided by the user. dmat[i][i] = 0 (self), dmat[i][j] = link cost, and typically 999 or a large number for no direct link.
  • Step 2: InitEach router starts with only direct link knowledge. rt[i].dist[j] = dmat[i][j] means "I know about direct neighbors." rt[i].from[j] = j means "to reach j, go directly to j."
  • Step 3: i loopSource router i — We're updating the routing table of router i. "What's the best path for i?"
  • Step 3: j loopDestination j — For each destination, we check if there's a better path.
  • Step 3: k loopIntermediate k — "Can I reach j cheaper by going through k?" dmat[i][k] = cost i→k (direct link), rt[k].dist[j] = k's best known path to j.
  • Bellman-FordUpdate condition — If the new path is shorter, update both the distance and the next-hop. Increment count so we know an update happened and need another pass.
  • UNTIL count==0Convergence — The do-while loop repeats until no routing table entry changes in a full pass. This guarantees the algorithm found the globally optimal routes.
  • Step 4Output — Prints routing table per router. Output shows: destination node, next-hop router (via which neighbor to leave), and total cost.
📝 30-MARK EXAM SUMMARY
DVR algorithm: Initialize routing tables from cost matrix → Triple nested loop (i=src, j=dst, k=intermediate) → Apply Bellman-Ford: if rt[i].dist[j] > dmat[i][k] + rt[k].dist[j] then update dist and from → Repeat until convergence (count==0) → Print tables. Time complexity: O(n³) per iteration. Uses do-while to guarantee at least one pass before checking convergence.
DVR — Tricks to Remember
🧠 THE MAGIC LINE
rt[i].dist[j] > dmat[i][k] + rt[k].dist[j]

Read as: "Current best distance from i→j" > "Cost of i→k (direct link)" + "k's best distance to j". If true → we found a shortcut via k!
🔄 WHY do-while?
We repeat until nothing changes (count == 0). This is convergence. Each iteration, more optimal paths propagate through the network. DVR is slow to converge — that's actually a known disadvantage!
📋 STRUCT TRICK
rt[i].dist[j] = shortest distance from node i to node j
rt[i].from[j] = which neighbor to use to get to j from i
💡 EXAM Q: What is the time complexity?
O(n³) per iteration × multiple iterations. Three nested loops over n nodes = O(n³). This is why DVR doesn't scale well to large networks.

#include <stdio.h>

void main(){

    // Variables:
    // in        -> Incoming packet size
    // out       -> Outgoing transmission rate
    // bucksize  -> Maximum bucket capacity
    // n         -> Number of packet inputs
    // store     -> Current data stored inside bucket
    int in, out, bucksize, n, store = 0;

    // Taking bucket configuration from user
    printf("Enter Bucket Size, Outgoing Rate and No of inputs\n");
    scanf("%d%d%d", &bucksize, &out, &n);

    // Process each incoming packet
    while(n > 0){

        // Read incoming packet size
        printf("Enter Incoming Packet Size\n");
        scanf("%d", &in);

        // Check whether packet can fit inside bucket
        if(in <= bucksize - store){

            // Store packet inside bucket
            store += in;

            // Display remaining free space
            printf("Available Size: %d\n", bucksize - store);

        }else{

            // If bucket overflows, drop extra packets
            printf("Dropped %d packets due less availabilty\n",
                   in - (bucksize - store));

            // Bucket becomes full
            store = bucksize;

            // No free space left
            printf("Available Size: %d\n", bucksize - store);
        }

        // Simulate outgoing transmission from bucket
        store -= out;

        // Prevent negative storage value
        if(store < 0)
            store = 0;

        // Display free space after outgoing packets leave
        printf("Available Size after Outgoing: %d\n",
               bucksize - store);

        n--;
    }
}

Algorithm covers: Purpose · Analogy · Data Structures · Core Logic · Full Pseudocode · Step-by-step explanation
Purpose
The Leaky Bucket Algorithm is a congestion control and traffic shaping mechanism. It converts bursty, irregular incoming traffic into a smooth, constant-rate outgoing stream. The bucket represents a buffer — packets fill it from the top (incoming), while it drains at a fixed rate from the bottom (outgoing). Packets that overflow the bucket are dropped.
Data Structures & Variables
  • bucksizeint — Maximum capacity of the bucket (buffer). No more than bucksize units can be stored at any time.
  • storeint — Current amount of data in the bucket. Starts at 0. Increases with incoming packets, decreases with outgoing drain.
  • inint — Size of the incoming packet in this round. Read from user input.
  • outint — Constant outgoing drain rate per round. Fixed — does not vary with input. This is the "leak."
  • nint — Number of input rounds (packets) to simulate. Controls the while loop.
Core Decision Logic
ACCEPT OR DROP?
Available space = bucksize - store

If in ≤ bucksize - store → Packet fits → Accept: store += in
Else → Overflow → Drop in - (bucksize - store) units → store = bucksize

After every round: Drain: store -= out. If store < 0, clamp to 0.
Full Pseudocode
BEGIN LeakyBucket // Step 1: Read configuration READ bucksize, out, n store ← 0 // Step 2: Process n incoming packets WHILE n > 0: READ in // incoming packet size this round // Step 3: Check if packet fits in bucket IF in ≤ (bucksize - store): store ← store + in // Accept all incoming data print "Available: " + (bucksize - store) ELSE: // Bucket would overflow dropped ← in - (bucksize - store) print "Dropped: " + dropped store ← bucksize // Fill bucket to max print "Available: 0" // Step 4: Drain the bucket at constant outgoing rate store ← store - out // Step 5: Prevent negative store (drain more than bucket held) IF store < 0: store ← 0 print "Available after outgoing: " + (bucksize - store) n ← n - 1 END LeakyBucket
  • Step 1Configuration — Three inputs: bucket capacity, constant drain rate per round, and number of rounds. store = 0 means bucket starts empty.
  • Step 3: AcceptAvailable space = bucksize - store. If the incoming packet fits entirely within the free space, accept all of it. Store increases by in.
  • Step 3: DropOverflow handling. The amount that exceeds available space is dropped: dropped = in - (bucksize - store). The bucket fills to max (store = bucksize). Only the excess is dropped, not the entire packet.
  • Step 4: DrainConstant-rate outgoing. Every round, out units leave the bucket regardless of input size. This is the "leak" — it's always the same, creating smooth output traffic.
  • Step 5: ClampEdge case guard. If store < 0 (outgoing rate exceeded what's in bucket), clamp to 0. Without this, store could go negative, which is meaningless for a buffer.
  • Key PropertyOutput is always bounded by out. No matter how large the incoming burst, the network beyond the bucket only ever sees at most out units per round. This is traffic shaping.
📝 30-MARK EXAM SUMMARY
Leaky Bucket: Read (bucksize, out, n) → Loop n times → If in ≤ bucksize-store: accept (store+=in) → Else: drop excess, fill bucket → Drain: store-=out → Clamp store to 0 if negative → Print available space. Purpose: traffic shaping — converts bursty input to smooth constant-rate output. vs Token Bucket: token bucket allows occasional bursts; leaky bucket enforces strict output rate.
Leaky Bucket — Tricks
💧 THE ANALOGY
Bucket = buffer. Water in = incoming packets. Hole = outgoing rate. If too much water → overflow (drop packets). Hole always drains at same rate regardless of input.
🔢 DROP CALCULATION
Dropped = incoming - (buck_size - store). This is: "how many packets exceeded available space?"
📝 EXAM Q: Leaky vs Token Bucket
Leaky = fixed output rate always (strict). Token Bucket = can burst occasionally (flexible). Leaky drops excess; Token stores tokens to allow bursts.
⚠️ EDGE CASE
After outgoing, if store goes negative — we clamp store to 0 (if (store < 0) store = 0;) to handle this.

A parity bit is added to every data unit so the total number of 1s becomes even (even parity). Detects all single-bit errors. Detects burst errors only if total errors in each data unit is odd.

Between any two points on an Ethernet network, there can be no more than 5 network segments, 4 repeaters, and of those 5 segments, only 3 can be populated with devices.

Q104 What is baseband vs broadband transmission?
Multiplexing & Media ▶
  • Baseband — entire cable bandwidth consumed by a single signal
  • Broadband — signals sent on multiple frequencies, allowing multiple simultaneous signals
Q105 What is Framing?
Security & Misc ▶

In the Data Link Layer, framing separates messages by adding sender and destination addresses. The destination address defines where to go; the sender address helps the recipient acknowledge receipt.

Q106 What is Bit Stuffing?
Security & Misc ▶

Adding an extra 0 whenever five consecutive 1s follow a 0 in the data, so the receiver doesn't mistake the bit pattern 0111110 for a flag (delimiter). Receiver removes the stuffed bit.

Q107 What is Character (Byte) Stuffing?
Security & Misc ▶

When a data section contains a byte with the same pattern as the flag, an escape character (ESC) is added before it. The receiver removes the ESC and treats the next byte as data, not a delimiter.

Q108 What is a Packet Filter?
Security & Misc ▶

A router with extra functionality that inspects every incoming/outgoing packet. Packets meeting defined criteria are forwarded; those that fail are dropped. Basic form of firewall.

Q109 What is Virtual Channel / Virtual Circuit?
Security & Misc ▶

A logical connection from one source to one destination (though multicast is possible). Also called a virtual channel. Provides the appearance of a dedicated path even over a shared network.

Q110 What is Virtual Path?
Security & Misc ▶

Along a transmission path from source to destination, a group of virtual circuits can be grouped together into a virtual path.

Q111 What is a Region (in hierarchical routing)?
Security & Misc ▶

In hierarchical routing, routers are divided into regions. Each router knows full detail about routing within its own region but nothing about the internal structure of other regions. Reduces routing table size.

Q112 What is Kerberos?
Security & Misc ▶

An authentication service developed at MIT. Uses encryption to prevent intruders from discovering passwords and gaining unauthorized access. Based on a trusted third-party key distribution center.

Q113 What is RAID?
Security & Misc ▶

Redundant Array of Independent Disks — a method for providing fault tolerance by using multiple hard disk drives. Data is distributed across drives for redundancy and/or performance.

Q114 What is Beaconing?
Security & Misc ▶

A process that allows a network to self-repair. Stations on the ring notify others when they are not receiving transmissions. Used in Token Ring and FDDI networks.

Q115 What is MAU (Token Ring)?
Security & Misc ▶

In Token Ring, the hub is called Multistation Access Unit (MAU).

Q116 What is Passive Topology?
Security & Misc ▶

When computers on the network simply listen and receive the signal without amplifying it. Linear bus is an example of passive topology.

Q117 What is a Brouter?
Security & Misc ▶

A hybrid device combining features of both a bridge and a router. Can route specific protocols and bridge others.

Q118 What is Frame Relay? Which layer?
Security & Misc ▶

Frame relay is a packet switching technology operating at the Data Link Layer (Layer 2). Designed for WANs; faster than older X.25 but with less error checking.

Q119 What is Terminal Emulation (Telnet)? Which layer?
Security & Misc ▶

Telnet is also called terminal emulation. It belongs to the Application Layer. Allows a user to log in and control a remote computer.

Q120 What is the difference between Simulation and Emulation?
Security & Misc ▶
  • Simulation — mimics the outward appearance/behavior of a system in software
  • Emulation — mimics the actual cause/process. Usually hardware-based; self-contained and can be connected to a development environment
Q121 What is Network Throughput?
Security & Misc ▶

Usually represented as an average and measured in bits per second (bps) or packets per second. An important indicator of network performance and quality. High packet loss → lower throughput.

Q122 What are key, ciphertext, and plaintext?
Security & Misc ▶
  • Plaintext — original readable information
  • Ciphertext — result of encrypting plaintext using an algorithm (cipher) + key
  • Key — special knowledge used to encrypt/decrypt; without it the ciphertext is unreadable
  • Decryption — process of converting ciphertext back to plaintext
Q123 What is a Linear Block Code?
Security & Misc ▶

A block code where the XOR (modulo-2 addition) of any two valid codewords produces another valid codeword.

Q124 What are Cyclic Codes?
Security & Misc ▶

Special linear block codes with an extra property: if a codeword is cyclically shifted (rotated), the result is another valid codeword.

Q125 What is Redundancy (in error detection)?
Security & Misc ▶

Including extra bits in transmission solely for comparison at the destination. These bits add no information — their purpose is purely to enable error detection.
